# ui-conversion-landing-page


Arquivo sugerido: `ui-conversion-landing-page.md`

## 1. Objetivo da skill

Esta skill orienta o Claude Code a criar, revisar e melhorar **landing pages orientadas à conversão**, mantendo qualidade visual, clareza, persuasão ética, responsividade, acessibilidade, performance, privacidade e segurança.

O objetivo é criar páginas que conduzam o usuário a uma ação principal com clareza, sem poluição visual, sem confusão cognitiva e sem comprometer regras técnicas do projeto.

Esta skill deve ser usada em páginas cujo objetivo principal seja:

- gerar leads;
- solicitar orçamento;
- vender produto digital;
- vender serviço;
- captar contatos;
- agendar atendimento;
- gerar conversa no WhatsApp;
- apresentar uma oferta;
- divulgar um curso;
- divulgar um material;
- divulgar uma solução;
- converter visita em ação mensurável.

---

## 2. Regra principal

Toda landing page deve ter **uma ação principal clara**.

A página pode ter ações secundárias, mas elas não podem competir visualmente com o CTA principal.

Antes de criar ou alterar a landing page, o Claude Code deve identificar:

1. Qual é a promessa principal?
2. Quem é o público?
3. Qual problema a página resolve?
4. Qual ação o usuário deve realizar?
5. Qual prova sustenta a promessa?
6. Quais objeções precisam ser respondidas?
7. Qual informação legal ou de privacidade precisa estar visível?
8. Qual fluxo acontece após o clique?
9. Quais estados o formulário ou CTA precisa ter?
10. Quais regras de segurança não podem ser alteradas?

---

## 3. Prioridade absoluta

Landing page boa não é apenas bonita. Ela precisa ser segura, acessível, clara e honesta.

A prioridade deve ser sempre:

1. Segurança
2. Privacidade / LGPD
3. Acessibilidade
4. Performance
5. Manutenibilidade
6. Clareza da oferta
7. Experiência do usuário
8. Conversão
9. Estética visual

Se uma decisão de conversão entrar em conflito com segurança, privacidade, acessibilidade ou transparência, ela deve ser rejeitada.

## 5. Quando não usar de forma rígida

Não transformar toda página do site em landing page agressiva.

Não aplicar esta skill de forma rígida em:

- páginas de política de privacidade;
- termos de uso;
- documentação técnica;
- artigos;
- páginas institucionais profundas;
- dashboards;
- áreas administrativas;
- sistemas internos;
- páginas cujo objetivo é informação, não conversão imediata.

Nesses casos, usar princípios de clareza e CTA contextual, mas sem forçar estrutura de venda.

## 6. Estrutura recomendada de landing page

Uma landing page de conversão deve seguir uma estrutura clara.

Estrutura base:

```text
1. Hero section
2. Prova social inicial
3. Benefícios principais
4. Como funciona
5. Diferenciais
6. Demonstração, mockup ou visual do produto/serviço
7. Depoimentos, cases ou credenciais
8. Oferta ou chamada de ação
9. FAQ
10. CTA final
11. Rodapé com informações legais
```

Nem toda landing page precisa de todas as seções.

O Claude Code deve adaptar ao contexto, evitando excesso.

## 7. Hero section

O hero é a primeira área de impacto.

Ele deve responder rapidamente:

- o que é oferecido;
- para quem é;
- qual benefício principal;
- qual ação deve ser tomada;
- por que confiar.

### 7.1 Elementos essenciais do hero

```text
Logo / identificação
Promessa principal
Subtítulo
Prova social curta
CTA principal
CTA secundário, se necessário
Imagem, mockup, formulário ou elemento visual de apoio
Microcopy de confiança
```

### 7.2 Promessa principal

A promessa deve ser específica.

Exemplo fraco:

```text
Soluções para você
```

Exemplo melhor:

```text
Controle seus atendimentos, sessões e pagamentos em uma única planilha pronta para usar
```

Exemplo fraco:

```text
Transporte com qualidade
```

Exemplo melhor:

```text
Transporte rodoviário de cargas líquidas com segurança, tecnologia e experiência nacional
```

### 7.3 Subtítulo

O subtítulo deve explicar a promessa sem repetir o título.

Deve responder:

- como funciona;
- por que importa;
- qual transformação entrega;
- qual público atende.

### 7.4 Prova social curta

No hero, a prova social deve ser curta e agrupada.

Exemplos:

```text
+ Pronto para Excel e Google Sheets
+ Campos organizados para rotina clínica
+ Fácil de adaptar
```

```text
+ Desde 1966
+ Operação nacional
+ Certificações reconhecidas
```

### 7.5 CTA principal

O CTA deve ser direto.

Exemplos:

```text
Solicitar orçamento
Falar pelo WhatsApp
Quero acessar o material
Agendar avaliação
Ver planos
Comprar agora
Baixar modelo
```

Evitar quando possível:

```text
Enviar
Clique aqui
Saiba mais
OK
```

### 7.6 CTA secundário

Usar CTA secundário apenas se fizer sentido.

Exemplos:

```text
Ver como funciona
Conhecer os benefícios
Ver demonstração
Tirar dúvidas
```

O CTA secundário não deve competir com o principal.

## 8. Padrão visual recomendado

Landing pages geralmente devem usar o padrão Z no hero.

### 8.1 Estrutura visual padrão Z

```text
Topo esquerdo:
- logo

Topo direito:
- navegação curta ou CTA secundário

Centro esquerdo:
- promessa
- subtítulo
- prova social
- CTA principal

Centro direito:
- imagem, mockup, formulário ou visual de apoio

Fechamento:
- microcopy de confiança
```

### 8.2 Mobile

No mobile, o fluxo deve virar vertical:

```text
1. Logo
2. Promessa
3. Subtítulo
4. Prova social
5. CTA principal
6. Visual ou formulário
7. Benefícios
8. CTA reforçado
```

O CTA deve aparecer cedo, mas não antes de o usuário entender a promessa.

---

## 9. Benefícios

Benefícios devem explicar o valor percebido.

Benefício não é apenas recurso.

### 9.1 Recurso x benefício

Recurso:

```text
Possui gráficos financeiros.
```

Benefício:

```text
Veja rapidamente entradas, despesas e saldo sem precisar montar fórmulas do zero.
```

Recurso:

```text
Frota rastreada.
```

Benefício:

```text
Mais controle e segurança durante toda a operação de transporte.
```

### 9.2 Estrutura de benefício

Cada benefício deve ter:

- título curto;
- explicação objetiva;
- ligação com dor ou desejo do público;
- se possível, um resultado claro.

### 9.3 Quantidade

Evitar excesso.

Recomendação:

- 3 benefícios no hero ou logo após;
- 4 a 6 benefícios na seção principal;
- mais que isso apenas se houver necessidade real.

---

## 10. Como funciona

A seção “Como funciona” reduz incerteza.

Deve explicar o processo em etapas.

Exemplo:

```text
1. Preencha seus dados
2. Receba o contato da equipe
3. Entenda a melhor solução
4. Inicie sua operação com segurança
```

Para produto digital:

```text
1. Acesse o material
2. Abra no Excel ou Google Sheets
3. Preencha seus dados
4. Acompanhe resultados com mais clareza
```

### 10.1 Regras

- usar poucas etapas;
- evitar texto longo;
- mostrar sequência lógica;
- manter números ou ícones consistentes;
- não prometer algo que não acontece;
- não esconder custos, prazos ou limitações.

---

## 11. Prova social e autoridade

Prova social deve reduzir desconfiança.

Tipos:

- depoimentos;
- logos de clientes;
- certificações;
- anos de atuação;
- números reais;
- cases;
- avaliações;
- prints, quando autorizados;
- selos;
- programas reconhecidos;
- histórico institucional.

### 11.1 Regras

- não inventar números;
- não inventar clientes;
- não inventar depoimentos;
- não usar logos sem autorização;
- não exagerar resultados;
- não usar prova falsa;
- não manipular informação;
- separar claramente prova real de argumento comercial.

Se não houver prova social real, usar credenciais, clareza do produto, demonstração ou garantias honestas.

---

## 12. Oferta

A oferta deve deixar claro:

- o que está incluído;
- para quem é;
- como acessar;
- o que acontece depois;
- valor, se aplicável;
- condições, se aplicáveis;
- limitações, se aplicáveis;
- suporte, se aplicável.

### 12.1 Regras

Não esconder:

- preço;
- prazo;
- renovação;
- condições;
- limitações;
- regras;
- informações legais.

Não usar dark patterns, como:

- falsa escassez;
- urgência inventada;
- contagem regressiva falsa;
- depoimentos falsos;
- botão enganoso;
- aceitar termos sem clareza;
- esconder cancelamento;
- induzir consentimento.

---

## 13. Formulários de landing page

Formulários devem ser simples, claros e seguros.

### 13.1 Campos

Solicitar apenas dados necessários.

Campos comuns:

- nome;
- e-mail;
- telefone;
- empresa, quando necessário;
- cidade, quando necessário;
- tipo de serviço;
- mensagem.

Evitar pedir dados sensíveis sem motivo.

### 13.2 Regras obrigatórias

- label visível;
- placeholder apenas como apoio;
- campos obrigatórios indicados;
- validação visual;
- validação segura;
- erro próximo do campo;
- loading ao enviar;
- sucesso após envio real;
- erro seguro se falhar;
- consentimento quando necessário;
- política de privacidade acessível;
- botão bloqueado durante envio quando necessário;
- proteção contra clique duplo.

### 13.3 Microcopy de privacidade

Exemplo:

```text
Usaremos seus dados apenas para retornar seu contato, conforme nossa Política de Privacidade.
```

### 13.4 Integração com WhatsApp

Quando o formulário abrir WhatsApp:

- informar que o WhatsApp será aberto;
- gerar mensagem clara;
- codificar a mensagem corretamente;
- não incluir dados sensíveis desnecessários;
- permitir o usuário revisar antes de enviar no WhatsApp;
- validar campos antes de abrir;
- não enviar automaticamente sem ação do usuário.

---

## 14. CTA para WhatsApp

O CTA para WhatsApp precisa ser claro.

Exemplos:

```text
Falar pelo WhatsApp
Solicitar orçamento pelo WhatsApp
Enviar informações para atendimento
Chamar equipe comercial
```

Evitar:

```text
Clique aqui
Enviar
Contato
```

### 14.1 Mensagem pré-preenchida

A mensagem deve ser objetiva.

Exemplo:

```text
Olá, gostaria de solicitar um orçamento. 
Nome:
Empresa:
Cidade:
Tipo de serviço:
Mensagem:
```

### 14.2 Segurança e privacidade

- não colocar CPF, documentos ou dados sensíveis sem necessidade;
- não expor dados na URL além do necessário;
- informar o usuário sobre a abertura do WhatsApp;
- respeitar consentimento quando houver coleta.

---

## 15. FAQ

FAQ serve para responder objeções.

Perguntas comuns:

- Para quem é?
- Como funciona?
- O que está incluso?
- Posso usar no celular?
- Posso usar no Excel ou Google Sheets?
- Como recebo o acesso?
- Como solicito orçamento?
- Em quanto tempo recebo retorno?
- Meus dados estão seguros?
- Posso falar com alguém antes?

### 15.1 Regras

- perguntas reais;
- respostas objetivas;
- não esconder informação importante;
- não usar FAQ para manipular;
- incluir privacidade quando houver formulário;
- não prometer suporte inexistente.

---

## 16. Rodapé

O rodapé deve conter informações importantes e legais.

Pode incluir:

- nome da empresa;
- CNPJ, se aplicável;
- contato;
- e-mail;
- telefone;
- endereço, se aplicável;
- links legais;
- política de privacidade;
- termos de uso;
- direitos autorais;
- redes sociais.

### 16.1 Regras

- não esconder política de privacidade;
- links devem ser clicáveis e reconhecíveis;
- contraste adequado;
- mobile legível;
- informações verdadeiras;
- não inventar dados.

---

## 17. Copywriting ético

A landing page deve ser persuasiva, mas honesta.

### 17.1 Permitido

- destacar benefícios reais;
- mostrar diferenciais reais;
- usar prova social verdadeira;
- explicar transformação possível;
- responder objeções;
- criar urgência real quando houver;
- mostrar clareza e confiança.

### 17.2 Proibido

- falsa escassez;
- promessa impossível;
- garantia enganosa;
- depoimento inventado;
- número inventado;
- manipulação de medo;
- esconder informação crítica;
- induzir clique enganoso;
- mascarar custo;
- dificultar cancelamento;
- pré-marcar consentimento sensível.

---

## 18. Design visual da landing page

O design deve ser:

- limpo;
- profissional;
- responsivo;
- coerente com a marca;
- visualmente hierárquico;
- fácil de escanear;
- focado na ação principal;
- sem poluição.

### 18.1 Evitar

- muitos botões competindo;
- muitos estilos diferentes;
- excesso de cores;
- excesso de efeitos;
- animações pesadas;
- texto pequeno;
- cards apertados;
- hero poluído;
- mockup irrelevante;
- formulário longo demais;
- elementos decorativos que atrapalham leitura.

### 18.2 Usar

- boa hierarquia;
- espaçamento generoso;
- contraste adequado;
- CTA forte;
- prova social agrupada;
- grids simples;
- cards consistentes;
- imagens otimizadas;
- microinterações leves;
- design system mínimo.

---

## 19. Preservação de layouts existentes

Quando já existir uma landing page, layout, componente, formulário, seção ou fluxo criado, o Claude Code não deve apagar tudo e refazer do zero automaticamente.

Antes de alterar a página, deve:

1. analisar a estrutura atual;
2. identificar a promessa atual;
3. identificar CTA principal;
4. identificar formulário e integrações;
5. identificar scripts, validações e eventos;
6. identificar estilos existentes;
7. identificar problemas visuais e de conversão;
8. sugerir melhorias por partes;
9. preservar funcionalidades existentes;
10. preservar integrações;
11. preservar validações;
12. preservar LGPD e links legais;
13. preservar classes, IDs e hooks importantes;
14. evitar refatoração completa sem justificativa clara.

### 19.1 Melhorias incrementais recomendadas

1. ajustar promessa;
2. melhorar subtítulo;
3. reposicionar CTA;
4. aproximar prova social da promessa;
5. reduzir campos do formulário, se possível;
6. melhorar texto dos botões;
7. adicionar loading/erro/sucesso;
8. melhorar FAQ;
9. ajustar mobile;
10. revisar links legais;
11. melhorar performance visual.

### 19.2 Proibição

Não substituir a landing page inteira por outra do zero apenas por estética.

Evoluir o que já existe com segurança e controle.

## 20. Segurança

A landing page deve respeitar as skills de segurança existentes.

### 20.1 Regras obrigatórias

- não remover sanitização;
- não remover validação backend;
- não remover proteção contra spam;
- não remover consentimento;
- não esconder política de privacidade;
- não expor tokens;
- não expor chaves de API;
- não usar scripts externos desnecessários;
- não adicionar dependência suspeita;
- não enviar dados sem ação clara do usuário;
- não mostrar erro técnico sensível;
- não permitir clique duplo problemático;
- não confiar apenas no frontend;
- não usar formulário sem tratamento de erro.

### 20.2 Dependências

Não instalar biblioteca de landing page, animação, formulário ou carrossel sem necessidade real.

Preferir:

- HTML semântico;
- CSS nativo;
- JavaScript simples;
- recursos já existentes;
- componentes do projeto.

### 20.3 Scripts externos

Evitar scripts externos desnecessários.

Quando houver scripts de analytics, pixels ou tags:

- respeitar consentimento;
- não duplicar tags;
- não vazar dados sensíveis;
- preservar LGPD;
- seguir configuração existente do projeto.

---

## 21. Privacidade e LGPD

Landing pages frequentemente coletam dados.

### 21.1 Regras

- informar finalidade da coleta;
- pedir apenas dados necessários;
- disponibilizar política de privacidade;
- usar consentimento quando necessário;
- não pré-marcar consentimento sensível;
- não enviar dados para terceiros sem clareza;
- não expor dados em URL sem necessidade;
- não armazenar dados sem necessidade;
- não usar rastreamento sem respeitar consentimento quando aplicável.

### 21.2 Texto de apoio

Exemplo:

```text
Ao enviar, você concorda que entraremos em contato usando os dados informados. Consulte nossa Política de Privacidade.
```

Adaptar ao contexto jurídico e ao projeto.

---

## 22. Acessibilidade

Landing page precisa ser acessível.

### 22.1 Regras

- heading principal único e claro;
- estrutura de headings lógica;
- contraste adequado;
- botões com foco visível;
- links reconhecíveis;
- labels nos campos;
- mensagens de erro associadas;
- navegação por teclado;
- textos alternativos em imagens relevantes;
- não depender apenas de cor;
- não depender apenas de animação;
- área de toque adequada;
- mobile legível.

### 22.2 CTA acessível

Botões devem ter texto claro.

Exemplo ruim:

```text
→
```

Exemplo melhor:

```text
Solicitar orçamento →
```

---

## 23. Performance

Landing page precisa carregar rápido.

### 23.1 Regras

- otimizar imagens;
- evitar vídeos pesados no hero;
- evitar animações contínuas;
- evitar bibliotecas desnecessárias;
- evitar carrossel pesado;
- evitar fontes em excesso;
- usar lazy loading quando adequado;
- evitar scripts duplicados;
- evitar excesso de DOM;
- manter CSS organizado.

### 23.2 Imagens

Imagens devem:

- ter tamanho adequado;
- usar compressão;
- ter `alt` quando informativas;
- não quebrar layout no mobile;
- não atrasar o CTA principal sem necessidade.

---

## 24. Estados de conversão

Toda landing page com ação deve prever estados.

### 24.1 CTA simples

Estados:

```text
default
hover
focus
active
disabled
loading
success
error
```

### 24.2 Formulário

Estados:

```text
initial
partially_filled
validating
submitting
success
validation_error
network_error
server_error
```

### 24.3 WhatsApp

Estados:

```text
initial
validating
generating_message
ready_to_open
opened
error
```

### 24.4 Regras

- não mostrar sucesso antes da ação real;
- não mostrar sucesso e erro juntos;
- não deixar usuário sem feedback;
- não apagar dados preenchidos após erro;
- bloquear clique duplo quando necessário.

- performance for adequada;
- nenhuma regra de segurança for enfraquecida;
- nenhuma dependência desnecessária for adicionada;
- layouts existentes forem preservados sempre que possível.

## 32. Resumo operacional para o Claude Code

Ao criar ou revisar uma landing page:

1. Identificar objetivo da conversão.
2. Definir promessa principal.
3. Definir CTA principal.
4. Definir público e dor.
5. Organizar hero com padrão Z.
6. Agrupar prova social perto da promessa.
7. Criar benefícios claros.
8. Explicar como funciona.
9. Responder objeções com FAQ.
10. Criar formulário simples e seguro.
11. Mapear estados de CTA e formulário.
12. Garantir LGPD e privacidade.
13. Garantir acessibilidade.
14. Garantir performance.
15. Preservar layout existente.
16. Melhorar por etapas.
17. Executar revisão final.

## Conexão com skills de segurança e do vault

Esta skill complementa — e nunca substitui — as camadas de segurança, privacidade e preservação.
